## 2. QUERYING Gap Analysis (the decisive gap)

### What run_query does in mcp-engine

Arbitrary DAX execution against a **connected** model (Desktop or Service) with: row limiting, query-plan capture, SE/FE engine timings, multi-run benchmarking with cache clearing, xEvents traces, VertiPaq storage stats, and RLS testing via `effective_user`/`effective_roles`.

### What pbip-tools has today

- **Library-level arbitrary DAX**: `executeQueries()` (fabric-client/datasets.ts:44-86) POSTs to the Power BI REST executeQueries endpoint — proven plumbing (auth, retry, 401 handling, redaction), **no registered tool**. `pbip_live_run_dax` is comment-only future work (live-list-model.ts:7), explicitly PAUSED pending a service-principal blast-radius verification (docs/PHASE_B_STATUS.md:7).
- **One live consumer**: `pbip_live_list_model` — 5 fixed `EVALUATE INFO.*()` queries for deployed-model schema snapshots (live-list-model.ts:173-179). Not exposed by the session's server build.
- **Offline**: `pbip_validate_dax` — 6 lexical checks + static ~354-function catalog; syntax only, never sees the model.
- **Auth**: service-principal `client_credentials` only (auth.ts:65-70). No delegated flow.

### GA-only implementation options (with cited limits)

**Option A — ship `pbip_live_run_dax` on the GA executeQueries endpoint. RECOMMENDED.**
Endpoint: `POST api.powerbi.com/v1.0/myorg/groups/{ws}/datasets/{ds}/executeQueries` — GA, works on Pro/PPU/Premium/Fabric (learn.microsoft.com/en-us/rest/api/power-bi/datasets/execute-queries, "Limitations", accessed 2026-07-13). Binding limits to encode in the tool description and schema:

- 1 query/call, 1 table/query, **max 100,000 rows OR 1,000,000 values** (whichever first), 15 MB/query, **120 requests/min/user** (verified verbatim).
- **DAX only — "MDX, INFO functions and DMV queries are not supported"** (verified). ⚠️ This directly conflicts with `pbip_live_list_model`'s INFO.\* design — **empirically verify against a real workspace before offboarding; the schema-snapshot tool may be broken on the GA endpoint** [doc and repo behavior conflict — flagged in inputs].
- Tenant settings: "Dataset Execute Queries REST API" + "Allow service principals to use Power BI APIs"; dataset Read+Build; **service principals NOT supported on RLS/SSO datasets regardless of settings** (verified).
- `maxRows` = trivial TOPN wrap (datasets.ts:36-38 already delegates this to the tool layer).

**Option B — RLS testing (effective_user half only).**
The GA endpoint accepts `impersonatedUserName` (verified) — threading it is ~5 lines — but it's dead under SP-only auth for RLS datasets (the exact datasets you'd test). Real cost = adding a user-delegated auth flow (moderate-high, changes the credential model). `effective_roles` has **no REST equivalent at all** (request body = queries/impersonatedUserName/serializerSettings, verified) — role-explicit testing is XMLA-only. **Decision: accept the loss; verify RLS with static role review (pbip_get_role) + "View as role" in Desktop/Service.**

**Option C — Arrow executeDaxQueries endpoint. DO NOT BUILD ON IT YET.**
`POST .../executeDaxQueries` allows INFO functions, `roles[]`, `effectiveUsername`, no fixed row cap, and would unlock REST-based VertiPaq stats via `INFO.STORAGETABLECOLUMNSEGMENTS()` (function GA in DAX; learn.microsoft.com/en-us/dax/info-storagetablecolumnsegments-function-dax). But the endpoint is **PREVIEW** (announcement blog titled "(Preview)"; no GA announcement found as of 2026-07-13; learn.microsoft.com/en-us/power-bi/developer/execute-dax-queries-arrow/overview) — under the workspace GA-only rule, track it as the future upgrade path and re-check GA status before any dependency.

**Option D — XMLA. OUT OF ARCHITECTURE.**
XMLA read is GA and default-on for Premium/PPU/Fabric (learn.microsoft.com/en-us/fabric/enterprise/powerbi/service-premium-connect-tools), but Microsoft's supported client libraries are .NET-only (ADOMD.NET/AMO-TOM/MSOLAP; learn.microsoft.com/en-us/analysis-services/client-libraries) — no supported Node path. Would require a .NET sidecar. Reject for pbip-tools.

**Note:** the Fabric REST API (api.fabric.microsoft.com) has **no** query-execution operation — SemanticModel items are CRUD-only (verified, learn.microsoft.com/en-us/rest/api/fabric/semanticmodel/items) — so queries stay on api.powerbi.com.

## 4. Cannot Replicate (accept these losses — decisively)

Architecturally out of reach for a Node static-file server + REST client. Do not attempt; keep external tooling for these.

1. **Power BI Desktop / local Analysis Services connectivity** — discover/select/reload local models, Desktop-target refresh, any live-Desktop interrogation. .NET AMO/TOM-only per Microsoft's client-library enumeration; no supported Node path. _Offboarding story: edit PBIP files (the git source of truth), open/reload in Desktop manually._
2. **DAX query plans and SE/FE engine timings** — no REST surface exists (both executeQueries variants return rows/error only, verified); XMLA trace + .NET tooling only. _Keep DAX Studio for deep perf work._
3. **xEvents/Profiler trace sessions and cache-controlled benchmarking** (ClearCache is an XMLA command) — same architectural wall. Multi-run REST timing is wall-clock-polluted and rate-limited (120 req/min).
4. **Role-explicit RLS query testing (`effective_roles`)** — no roles param on the GA REST endpoint (verified); XMLA `Roles=` connection-string feature only. User-UPN impersonation is additionally blocked by SP-only auth on RLS datasets. _Mitigation: static role-DAX review + "View as role" in Desktop/Service._
5. **VertiPaq storage statistics via GA surfaces** — GA executeQueries blocks INFO/DMV (verified); the only REST path is the PREVIEW Arrow endpoint. Preview-gated until Microsoft GAs it; re-check then.
6. **DMV queries generally** — not supported on the GA REST endpoint; XMLA-only, with database-admin-level DMV ceiling even there.
7. **Per-operation undo of uncommitted state / live-TOM transactional writes with instant engine validation** — the file-based loop (edit → validate_tmdl → git → deploy → refresh) is the intentional trade; bad DAX identifiers surface at deploy/refresh, not edit time. P0 #5 (pre-write backup) + git discipline is the ceiling here.
8. **Server-side policy enforcement across arbitrary MCP hosts** — pbip-tools delegates to the host; acceptable in this single-user Claude Code deployment, a real gap only if the server is ever exposed to other clients.

**Bottom line:** offboarding is viable. The static-file surface (model/report/RLS authoring, auditing, BPA, dependencies, RDL) is at or beyond mcp-engine parity and is the better fit for this git-first workflow. The one load-bearing build is **P1 `pbip_live_run_dax` on the GA executeQueries endpoint**; the four P0 round-trip fixes are non-negotiable before trusting writes on real models; Desktop-TOM, query plans, and role-explicit RLS testing are permanent losses to cover with Desktop/DAX Studio outside the MCP loop.

## Addendum: decisions taken 2026-07-13 (same session)

1. **Data-boundary rationale made explicit:** offboarding mcp-engine is driven by data sovereignty (no work data through a third-party MCP server), not just capability parity. Egress audit of pbip-tools found exactly one non-Microsoft host: `www.daxformatter.com`.
2. **Remote formatter REMOVED (supersedes the "formatter gate" idea):** `pbip_format_dax` + `pbip_format_measures` deregistered from the MCP surface (commit `3bc7459`). Server egress is now Microsoft endpoints only, by construction. `formatDax`/`formatDaxBatch` remain as deliberate library imports; `pbip_validate_dax` (offline) unaffected. Rationale: formatting is redundant in an agent-driven workflow, and the remote syntax-oracle value is superseded once `pbip_live_run_dax` (P1 #7) ships.
3. **GA-only rule:** layered approach agreed in principle — GA executeQueries as default path; Arrow preview endpoint only behind a preview-flagged adapter; global CLAUDE.md carve-out for internal tooling still to be codified.
